# 架构与入口说明

## 1. 总体结构

项目采用“共享核心 + 多界面”的结构：

- `core/`
  统一封装音源搜索、批量匹配、下载逻辑、下载器适配和公共配置
- `pyqt_ui/`
  直接调用 `core/`，提供传统桌面窗口，定位为兼容入口
- `backend/`
  在 `core/` 之上暴露 Web API、任务状态和下载历史管理
- `frontend/`
  通过 `backend/` 的 API 提供 React Web 界面，是当前产品基线

## 2. 两套界面的关系

PyQt 和 Web 不是互相调用关系，而是并列关系：

- PyQt：本地窗口直接调 Python 逻辑
- Web：前端调 FastAPI，FastAPI 再调共享核心

所以“功能对齐”真正要看的不是页面长得一样，而是：

- 是否都能调用同样的核心能力
- Web 主界面是否完整可用
- 桌面端是否还能作为兼容入口访问共享核心

## 3. 当前推荐入口

优先使用 Web 版：

- 批量搜索
- 歌单导入
- 下载历史管理

桌面版保留用于：

- 兼容旧使用习惯
- 核对旧流程
- 本地单窗口使用

## 4. 启动入口

- `START.bat`
  默认启动 Web 主界面
- `START_WEB.bat`
  显式启动前端和后端
- `START_DESKTOP.bat`
  启动本地 PyQt 遗留窗口
- `START_DOCKER_DESKTOP.bat`
  启动 Docker 中的 PyQt 遗留桌面版

## 5. 当前基线

建议以后把以下项目视为“Web 主界面必须可用”的主流程：

- 批量搜索
- 歌单导入
- 下载
- 下载历史查看
- 打开文件夹
- 清理无效记录

不要求完全一致的部分：

- 桌面端特有的窗口交互细节
- Web 端特有的路由、页面布局、异步状态显示
- PyQt 保留的旧入口和工具性操作
